# Accessibility Skill

## 목적

모든 사용자가 서비스를 이용할 수 있도록 접근성 기준을 자동 적용합니다.

## 활성화 조건

다음 상황에서 자동 활성화:
- 화면 기획서 작성 시
- UI 컴포넌트 설계 시
- 색상, 폰트 크기 지정 시
- "접근성", "WCAG", "스크린리더" 키워드 언급 시

## WCAG 2.1 AA 기준

### 1. 인지 가능 (Perceivable)

#### 1.1 대체 텍스트

**원칙**: 모든 비텍스트 콘텐츠에 대체 텍스트 제공

**체크리스트**:
- [ ] 이미지: alt 속성 작성
- [ ] 아이콘: aria-label 제공
- [ ] 그래프/차트: 데이터 테이블 대안 제공

**예시**:

```markdown
❌ 나쁜 예:
[아이콘: ✓]

✅ 좋은 예:
[아이콘: ✓ (완료)]
Alt: "작업이 성공적으로 완료되었습니다"
```

#### 1.2 색상 대비

**원칙**: 텍스트와 배경의 명도 대비 4.5:1 이상

**체크리스트**:
- [ ] 일반 텍스트: 4.5:1 이상
- [ ] 큰 텍스트 (18pt+): 3:1 이상
- [ ] 중요 UI 요소: 3:1 이상

**예시**:

```markdown
❌ 나쁜 예:
- 배경: #FFFFFF (흰색)
- 텍스트: #CCCCCC (연한 회색)
- 대비: 1.6:1 ❌

✅ 좋은 예:
- 배경: #FFFFFF (흰색)
- 텍스트: #333333 (진한 회색)
- 대비: 12.6:1 ✅
```

**자동 체크**:
```
[ACCESSIBILITY] 색상 대비 부족

현재 대비: 2.1:1
필요 대비: 4.5:1 이상

제안:
- 텍스트를 더 어둡게: #757575 → #333333
- 또는 배경을 더 밝게
```

#### 1.3 색상만으로 정보 전달 금지

**원칙**: 색상 외 추가 단서 제공 (텍스트, 아이콘, 패턴)

**예시**:

```markdown
❌ 나쁜 예:
- 성공: 녹색 배경
- 오류: 빨간색 배경
(색맹 사용자는 구분 불가)

✅ 좋은 예:
- 성공: 녹색 배경 + ✓ 아이콘 + "성공"
- 오류: 빨간색 배경 + ✗ 아이콘 + "오류"
```

### 2. 운용 가능 (Operable)

#### 2.1 키보드 접근성

**원칙**: 모든 기능을 키보드로 조작 가능

**체크리스트**:
- [ ] 모든 버튼/링크: Tab 키로 포커스 이동
- [ ] 모달/팝업: Esc 키로 닫기
- [ ] 드롭다운: 화살표 키로 선택
- [ ] 포커스 표시: 명확한 아웃라인

**키보드 단축키 예시**:

| 동작 | 키 |
|------|-----|
| 다음 항목 | Tab |
| 이전 항목 | Shift + Tab |
| 선택/실행 | Enter 또는 Space |
| 닫기 | Esc |
| 메뉴 열기 | Alt + 문자키 |

#### 2.2 포커스 표시

**원칙**: 현재 포커스된 요소를 명확히 표시

**예시**:

```css
/* 기획서에 명시 */
포커스 스타일:
- 테두리: 2px solid #0066CC (파란색)
- 아웃라인: 2px offset
```

#### 2.3 충분한 클릭/터치 영역

**원칙**: 최소 44x44px (모바일), 24x24px (데스크톱)

**체크리스트**:
- [ ] 버튼: 최소 크기 준수
- [ ] 아이콘 버튼: 패딩 추가로 영역 확보
- [ ] 인접 요소: 최소 8px 간격

**예시**:

```markdown
❌ 나쁜 예:
[삭제] 아이콘 버튼: 16x16px

✅ 좋은 예:
[삭제] 아이콘: 16x16px + 패딩 14px = 44x44px 터치 영역
```

### 3. 이해 가능 (Understandable)

#### 3.1 명확한 레이블

**원칙**: 모든 입력 필드에 레이블 제공

**체크리스트**:
- [ ] 레이블: 입력 필드 위 또는 좌측
- [ ] Placeholder: 레이블 대신 사용 금지 (보조만)
- [ ] 필수 표시: 별표(*) + "필수" 텍스트

**예시**:

```markdown
❌ 나쁜 예:
[입력 필드]
Placeholder: "이름을 입력하세요"

✅ 좋은 예:
레이블: "이름 *필수"
[입력 필드]
Helper text: "실명을 입력해주세요"
```

#### 3.2 오류 식별 및 복구

**원칙**: 오류 발생 시 명확한 설명과 해결 방법 제공

**예시**:

```markdown
오류 메시지:
- 위치: 해당 입력 필드 바로 아래
- 색상: 빨간색 + 오류 아이콘
- 텍스트: "이메일 형식이 올바르지 않습니다. 예) user@example.com"
```

### 4. 견고성 (Robust)

#### 4.1 시맨틱 마크업

**원칙**: 적절한 HTML 태그 사용

**기획서에 명시**:

```markdown
## 화면 구조

- 헤더: <header> 태그
- 메인 콘텐츠: <main> 태그
- 네비게이션: <nav> 태그
- 버튼: <button> 태그 (div 금지)
- 폼: <form> + <label> + <input> 구조
```

## 체크리스트 (기획서 작성 시)

### UI 컴포넌트 설계

- [ ] **버튼**
  - 최소 크기: 44x44px (모바일), 24x24px (데스크톱)
  - 포커스 표시: 2px 파란색 아웃라인
  - 키보드 접근: Tab, Enter/Space

- [ ] **입력 필드**
  - 레이블 위치: 필드 위
  - 오류 메시지: 필드 아래, 빨간색
  - 포커스 표시: 테두리 색상 변경

- [ ] **모달/팝업**
  - Esc 키로 닫기
  - 첫 포커스: 제목 또는 첫 번째 버튼
  - 백그라운드: 반투명 검정 (0.5 opacity)

- [ ] **이미지/아이콘**
  - Alt 텍스트 작성
  - 장식용 이미지: alt="" (빈 값)

### 색상 설계

- [ ] 텍스트 대비: 4.5:1 이상
- [ ] UI 요소 대비: 3:1 이상
- [ ] 색상만으로 정보 전달 금지 (아이콘/텍스트 병행)

### 콘텐츠

- [ ] 명확한 제목 계층 (H1 > H2 > H3)
- [ ] 링크 텍스트: "여기" 금지, "상세보기" 사용
- [ ] 오류 메시지: 문제 + 해결 방법

## 자동 위반 감지

기획서 작성 시 다음을 자동 체크:

```
[ACCESSIBILITY] 위반 감지

❌ 버튼 크기: 20x20px (최소 24x24px 필요)
❌ 색상 대비: 2.8:1 (최소 4.5:1 필요)
⚠️ 이미지 alt 텍스트 누락

개선 제안:
1. 버튼 크기를 44x44px로 증가 (패딩 추가)
2. 텍스트 색상을 #333333으로 변경 (대비 12.6:1)
3. 이미지에 대체 텍스트 추가: "대시보드 그래프"
```

## 접근성 테스트 가이드

기획서에 포함할 테스트 항목:

```markdown
## 접근성 테스트 체크리스트

### 키보드 테스트
- [ ] 모든 기능을 마우스 없이 사용 가능
- [ ] Tab 순서가 논리적
- [ ] 포커스 표시가 명확

### 스크린리더 테스트
- [ ] VoiceOver (macOS/iOS) 테스트
- [ ] NVDA (Windows) 테스트
- [ ] 모든 콘텐츠가 음성으로 전달

### 색상/대비 테스트
- [ ] 명암 대비 확인 (도구: WebAIM Contrast Checker)
- [ ] 색맹 시뮬레이션 (도구: Colorblind Web Page Filter)

### 확대/축소 테스트
- [ ] 200% 확대 시 콘텐츠 잘림 없음
- [ ] 텍스트 크기 조정 시 레이아웃 유지
```
